🧭 วิธีเลือกอัลกอริทึม

อ่านโจทย์แล้วจับสัญญาณว่าต้องใช้อัลกอริทึมตระกูลไหน — ทักษะที่สำคัญที่สุดของการแข่ง/สัมภาษณ์ สำคัญกว่าการเขียนโค้ดเสียอีก

🪜 กรอบคิด 4 ขั้น (ใช้ได้กับทุกโจทย์)

1
โจทย์ถามอะไร?
หาค่าน้อย/มากสุด? · นับจำนวนวิธี? · เป็นไปได้ไหม/หาคำตอบสักอัน? · สร้าง/จัดเรียง?
2
ขนาดข้อมูล n เท่าไร?
บอก เป้าหมาย Big-O ที่ยอมรับได้ — สัญญาณที่ทรงพลังที่สุด (ดูตารางที่ 1)
3
ข้อมูลโครงสร้างแบบไหน?
กราฟ/เครือข่าย? · เรียงได้ไหม? · ต้นไม้? · ช่วงเวลา? · กริด? · ตัวเลข/จำนวนเฉพาะ? · จุด/รูปทรง?
4
มีคำใบ้อะไร?
keyword ในโจทย์ที่ชี้ตรง ๆ ไปยังตระกูลอัลกอริทึม (ดูตารางที่ 2)

📏 ตารางที่ 1 — ดู “ขนาด n” เดาเป้าหมาย Big-O

สมมติคอมพิวเตอร์ทำงานราว 10⁸ ครั้ง/วินาที — เห็นค่า n ก็เดาความซับซ้อนที่ต้องได้ และตัดอัลกอริทึมที่ช้าเกินทิ้งได้ทันที

n สูงสุดBig-O ที่ต้องได้อัลกอริทึมที่ใช่
≤ 11O(n!)Permutations, brute force
≤ 20–25O(2ⁿ)Backtracking, Bitmask DP
≤ 100–500O(n³)Floyd-Warshall, Matrix-Chain, interval DP
≤ 2,000–5,000O(n²)DP, Edit Distance
≤ 10⁵–10⁶O(n log n)Sort, Binary Search, Segment Tree, Dijkstra, Greedy
≤ 10⁷–10⁸O(n)Two Pointers, Sliding Window, Kadane
สูงมาก (10¹⁸)O(log n) / สูตรModular Exp, GCD, คณิตศาสตร์

💡 เห็น n ≤ 18 → แทบฟันธงเป็น bitmask/backtracking · เห็น n ≤ 10⁶ → ห้ามคิด O(n²) เด็ดขาด

🎯 ตารางที่ 2 — ดู “โจทย์ถามอะไร + คำใบ้” เดาตระกูลอัลกอริทึม

สัญญาณในโจทย์อัลกอริทึมที่น่าจะใช่
“เส้นทาง/ต้นทุนน้อยที่สุด” บนเครือข่ายDijkstra (น้ำหนักบวก) · Bellman-Ford (ติดลบ) · BFS (ไร้น้ำหนัก)
“จำนวนก้าวน้อยสุดไปถึงสถานะหนึ่ง”BFS บน state graph
“มาก/น้อยสุด” + ปัญหาย่อยซ้อนทับ + “นับจำนวนวิธี”Dynamic Programming
“เลือกแบบโลภแล้วน่าจะดีสุด” (เรียงแล้วคว้า)Greedy (ต้องพิสูจน์ greedy-choice)
“เป็นไปได้ไหม / หาคำตอบสักชุด” + ลองแล้วถอยBacktracking
“ถามค่าในช่วง [l..r] + แก้ค่า” บ่อย ๆSegment Tree · Fenwick Tree
“กลุ่ม / เชื่อมต่อกันไหม / component”Union-Find · DFS/BFS
“จุดที่ขาดแล้วกราฟแยก”Articulation / Bridges
“ค่ามากสุดที่ยังผ่านเงื่อนไข” (monotonic)Binary search on answer
“คู่/ช่วงต่อเนื่องบนข้อมูลเรียงแล้ว”Two Pointers · Sliding Window
“หา pattern ในข้อความ”KMP · Rabin-Karp
“k-th น้อยสุด / มัธยฐาน”Quickselect · Heap
“สองผู้เล่นเล่นดีที่สุด / expected value ภายใต้การเลือกดีสุด”Minimax · ทฤษฎีเกม · value iteration

โจทย์จริงจากเวทีระดับโลก แต่ละข้อแสดง 3 ส่วน: 📋 เนื้อโจทย์ → 🧭 คิดเลือกอัลกอริทึม → ✅ เฉลย · ตัวเลข “แก้ได้กี่ทีม” (จาก ~140 ทีม) = ตัวบอกความยาก

🔦 Bridging the Gap — ICPC WF 2023 · แก้ได้ 1 ทีมเดียว

📋 เนื้อโจทย์

คน n คนต้องข้ามสะพานตอนกลางคืน สะพานรับน้ำหนักได้ครั้งละไม่เกิน c คน และมีไฟฉายเพียงอันเดียว (ต้องใช้ทุกครั้งที่ข้าม) กลุ่มที่ข้ามพร้อมกันต้องเดินช้าเท่าคนที่ช้าที่สุดในกลุ่ม จงหาเวลารวมน้อยที่สุดให้ทุกคนข้ามสะพานได้

ตัวอย่าง: 4 คน เวลาข้าม [1, 2, 5, 10], c = 2 → คำตอบ 17 นาที
(2 คนเร็วข้าม=2 · คนเร็วสุดกลับ=1 · 2 คนช้าข้าม=10 · คนเร็วรองกลับ=2 · 2 คนเร็วข้าม=2)
🧭 คิดเลือกอัลกอริทึมทีละขั้น
  1. ถามอะไร? หา “เวลารวมน้อยที่สุด” → ปัญหา optimization
  2. ขนาด n? n ≤ 10⁴ → รับ O(n²) ได้ ไม่ต้องรีบหาสูตร O(n)
  3. โครงสร้าง? ต้องตัดสินใจเป็นลำดับว่าใครข้ามไป/ใครพายไฟฉายกลับ — เหมือนเลือกแบบโลภ แต่ลำดับมีหลายแบบ
  4. คำใบ้? “เลือกกลุ่มข้ามแบบดีสุดแต่ละครั้ง” ส่งกลิ่น greedy — แต่ greedy ล้วนพิสูจน์ยากและพลาดได้ จึงต้องมี “สถานะ” มาช่วยคิด → DP
  5. → สรุป: Greedy (โครงสร้างคำตอบ) + DP (คำนวณต้นทุน)
✅ เฉลย

ข้อสังเกต: การพายไฟฉายกลับทำทีละ 1 คนเสมอ และการข้ามไปมีแค่ไม่กี่รูปแบบ (เช่น เอา k คนช้าสุด + c−k คนเร็วสุดที่จะพายกลับทีหลัง) → ตั้ง DP บนสถานะ (จำนวนคนช้าที่ข้ามไปแล้ว, จำนวนการพายกลับที่ “จ่ายล่วงหน้า” ไว้) เนื่องจากไม่จำเป็นต้องสะสมการพายกลับเกิน n/c ครั้ง สถานะจึงเหลือ O(n²/c) — เร็วพอสำหรับ n=10⁴

⏰ Jet Lag — ICPC WF 2023 · แก้ได้ 96 ทีม (ค่อนข้างง่าย)

📋 เนื้อโจทย์

มีกิจกรรม n อย่างในงาน แต่ละอย่างเป็นช่วงเวลา [bᵢ, eᵢ] ที่คุณต้องตื่นอยู่ตลอด กติกาการนอน: ถ้านอน k นาที จะ “อิ่มตัว” (ห้ามนอน) อีก k นาที แล้วจึงใช้งานได้อีก k นาที (จะนอนหรือตื่นก็ได้) เริ่มที่นาที 0 ต้องนอนทันที จงหาตารางนอนที่ไม่พลาดกิจกรรมใดเลย หรือบอกว่า impossible

🧭 คิดเลือกอัลกอริทึมทีละขั้น
  1. ถามอะไร? สร้างตาราง (construct) ที่ feasible หรือบอก impossible
  2. ขนาด n? n ≤ 2×10⁵ → ต้อง O(n) หรือ O(n log n) — DP หนัก ๆ ใช้ไม่ได้
  3. โครงสร้าง? เป็นปัญหาบน “ช่วงเวลา (interval)” เรียงตามเวลาอยู่แล้ว
  4. คำใบ้? interval + “เลือกช่วงนอนให้ตื่นได้นานที่สุด” = ลายเซ็นคลาสสิกของ greedy
  5. → สรุป: Greedy (โค้ดสั้นมาก)
✅ เฉลย

เริ่มด้วยการละเงื่อนไข “ห้ามนอนตอนอิ่มตัว” ไว้ก่อน → นอนในช่วงว่างระหว่างกิจกรรม [eᵢ, bᵢ₊₁] แบบโลภเฉพาะเมื่อมันทำให้ตื่นได้นานขึ้น จากนั้นค่อยปรับช่วงนอนที่ติดกันให้เว้นระยะอิ่มตัวให้ถูก (เลื่อนจุดจบการนอนเป็นค่าเฉลี่ยของจุดเริ่มสองช่วงเมื่อจำเป็น) — ไอเดีย greedy ที่ถูกต้องทำให้เฉลยทีมสั้นสุดเพียง 669 ไบต์!

🎲 Alea Iacta Est — ICPC WF 2023 · แก้ได้ 11 ทีม

📋 เนื้อโจทย์

เล่นเกมด้วยลูกเต๋า d ลูก (d ≤ 6) แต่ละหน้ามีตัวอักษร ทอยแล้วอยากเรียงตัวอักษรบนสุดให้เป็นคำในพจนานุกรม (มี w คำ) ถ้าทำไม่ได้ก็เลือกลูกที่จะทอยใหม่ (reroll) บางลูกได้ จงหาจำนวนครั้งการทอยที่คาดหวัง (expected) น้อยที่สุด เมื่อเล่นด้วยกลยุทธ์ที่ดีที่สุด

🧭 คิดเลือกอัลกอริทึมทีละขั้น
  1. ถามอะไร? หาค่า expected “น้อยที่สุด” ภายใต้การเลือก reroll ที่ดีที่สุด → optimize + ตัดสินใจดีสุด
  2. ขนาด n? d ≤ 6 → จำนวนสถานะของลูกเต๋า ≈ 7ᵈ ≤ 7⁶ ≈ 118,000 (เล็กพอจะมองเป็นโหนดได้)
  3. โครงสร้าง? แต่ละ “สถานะของลูกเต๋าทั้งหมด” (แต่ละลูกโชว์เลขอะไร หรือกำลัง reroll) = โหนด; การทอย/reroll = การย้ายสถานะที่มีต้นทุนและความน่าจะเป็น
  4. คำใบ้? “ไปถึงสถานะเป้าหมาย (เป็นคำ) ด้วยต้นทุนน้อยที่สุด” = shortest path บนกราฟสถานะ
  5. → สรุป: Dijkstra / Bellman-Ford (หรือ value iteration)
✅ เฉลย

มองสถานะเป็นเวกเตอร์ v ∈ {1..6, ∗}ᵈ โดย ∗ = ลูกที่กำลัง reroll · เป้าหมายคือ “fixed state” ที่จัดเป็นคำได้ · จาก fixed state เลือก reroll ลูกไหนก็ได้ (cost 0) ไปยัง “undetermined state” · จาก undetermined state การทอยให้ผลตามความน่าจะเป็น (เพิ่มจำนวนทอย +1) · expected cost ที่น้อยสุดจึงคือ shortest path บนกราฟนี้ → แก้ด้วย Dijkstra เวอร์ชันสำหรับ expected value (value iteration)

🦓 Zoo Management — ICPC WF 2022 · แก้ได้ 15 ทีม

📋 เนื้อโจทย์

สวนสัตว์มีคอก (node) เชื่อมกันด้วยอุโมงค์ (edge) ทุกคอกมีสัตว์อยู่เต็ม ย้ายสัตว์ได้โดยให้สัตว์หลายตัวเดินไปตามอุโมงค์พร้อมกัน (สัตว์เลื่อนไปคอกถัดไปตาม edge พร้อมกันหมด จึงไม่เจอกันในอุโมงค์ — สลับสองคอกที่ติดกันตรง ๆ ไม่ได้) ถาม: จัดสัตว์ให้เป็นรูปแบบเป้าหมายได้หรือไม่ (possible / ไม่ได้)

🧭 คิดเลือกอัลกอริทึมทีละขั้น
  1. ถามอะไร? เป็นไปได้ไหม (yes/no) ที่จะจัดสัตว์เป็น permutation เป้าหมาย
  2. โครงสร้าง? ข้อมูลคือกราฟ และการย้าย = หมุนสัตว์ไปตาม cycle ของกราฟ
  3. คำใบ้? edge ที่ไม่อยู่ในวงรอบ (cycle) ใด ๆ เลย = bridge → สัตว์หมุนผ่านมันไม่ได้ จึง “ตัดทิ้ง” ได้ แล้วแยกปัญหาเป็นชิ้น ๆ
  4. → สรุป: Bridges / Articulation ย่อยกราฟเป็น 2-edge-connected component แล้วแก้ทีละชิ้น
✅ เฉลย

ลบ bridge ทั้งหมดทิ้งได้เลย (ไม่มี cycle ผ่าน) เหลือเป็นปัญหาย่อยอิสระทีละ 2-edge-connected component · ถ้า component เป็น cycle เดี่ยว ๆ → เช็คว่าจัดเป็นเป้าหมายได้ไหมด้วยการเทียบสตริงแบบวน (string matching เชิงเส้น) · ถ้า component มีหลาย cycle → พิสูจน์ได้ว่าสร้าง “ทุก permutation” หรือ “เฉพาะ even permutation” ได้ ตามโครงสร้าง

💡 เคล็ดลับนักแข่ง: อ่าน constraint ก่อนอ่านโจทย์ให้จบด้วยซ้ำ — ตัวเลข n บอกครึ่งหนึ่งของคำตอบแล้ว จากนั้นค่อยจับ keyword ว่าโจทย์ “ถามอะไร” เพื่อเลือกตระกูลอัลกอริทึม
