หมวด 2 · โครงสร้างข้อมูลแบบต้นไม้

B+ Tree

แนวคิด

B+ Tree เป็น B-Tree เวอร์ชันที่ฐานข้อมูล/ระบบไฟล์ใช้จริง (MySQL InnoDB, PostgreSQL) ต่างจาก B-Tree ตรง:

  • คีย์จริงอยู่ที่ "ใบ" ทั้งหมด · โหนดภายในเก็บแค่ ตัวคั่น (separator) ไว้นำทาง (เป็นสำเนาของคีย์)
  • ใบทุกใบเชื่อมกันเป็นลิงก์ (→) ทำให้ สแกนช่วง (range scan) เร็วมาก แค่เดินตามลิงก์
  • เวลาใบเต็มแล้วแยก จะ คัดลอก (copy) คีย์กลางขึ้นไปเป็นตัวคั่น (คีย์ยังอยู่ที่ใบด้วย) — ต่างจาก B-Tree ที่ ย้าย ขึ้น

ความซับซ้อน: ค้นหา/แทรก O(log n) · range query มีประสิทธิภาพสูงเป็นพิเศษ

กำลังเดินผ่าน เต็ม → กำลังแยก คีย์ที่เพิ่งแทรก โหนดใบ (มีคีย์จริง + ลิงก์ →)

📝 ตัวอย่างโจทย์ & การนำไปใช้

โจทย์: ตาราง orders ใน MySQL ต้องรันคิวรี WHERE order_date BETWEEN '2024-01-01' AND '2024-03-31' เร็ว ๆ บนข้อมูลหลายล้านแถว

แก้ด้วย B+ Tree อย่างไร: ค้นหาขอบล่างของช่วงลงไปถึงใบ O(log n) จากนั้น เดินตามลิงก์ระหว่างใบ (leaf link) เก็บผลทีละแถวจนถึงขอบบน — range scan เร็วมากเพราะคีย์จริงอยู่ที่ใบทั้งหมดและเรียงต่อเนื่องกัน ไม่ต้องวนกลับขึ้นโหนดภายใน · fan-out สูง ต้นไม้เตี้ย อ่านดิสก์น้อย

เจอได้ที่ไหน: ดัชนีฐานข้อมูลที่ใช้จริง — MySQL InnoDB (clustered index ใช้ B+ Tree), PostgreSQL, Oracle · ระบบไฟล์ (NTFS, XFS), range query / ORDER BY / pagination
